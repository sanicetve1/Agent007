# 🧰 Step 2: Tool Categories for a Bank Underwriting Agent

Don’t think in APIs first.

Think in **decision capabilities**.

---

## 1️⃣ Data Retrieval Tools (Deterministic)

These should be pure, auditable services.

Examples:

* `get_credit_bureau_score(applicant_id)`
* `get_internal_transaction_history(customer_id)`
* `get_existing_loan_exposure(customer_id)`
* `get_collateral_details(asset_id)`
* `get_income_statements(document_id)`

These are not intelligent.
They are reliable.

They map to:

* Core banking system
* Credit bureau
* Data warehouse
* Document management system

---

## 2️⃣ Analytical Tools (Model-Based)

These introduce probabilistic reasoning but are still controlled.

Examples:

* `calculate_dti_ratio(data)`
* `run_pd_model(applicant_features)`
* `run_lgd_model(collateral_features)`
* `fraud_risk_score(transaction_pattern)`
* `affordability_score(income, obligations)`

These can wrap:

* ML models (your RandomForest background fits here 😉)
* XGBoost
* Internal scoring engines

Still deterministic from a system perspective.

---

## 3️⃣ Interpretive Tools (Agent Territory)

This is where agents shine.

Examples:

* `extract_income_from_pdf(document)`
* `summarize_bank_statement(document)`
* `classify_employment_type(text)`
* `detect_inconsistencies(application_text)`
* `explain_model_decision(model_output)`

These handle:

* Unstructured data
* Ambiguity
* Text reasoning

This is where LLM reasoning adds real value.

---

## 4️⃣ Governance / Rule Enforcement Tools

These protect you.

Examples:

* `check_regulatory_mandatory_steps(application_state)`
* `validate_kyc_status(customer_id)`
* `verify_aml_screening(customer_id)`
* `enforce_policy_thresholds(score)`

These should NOT be agent-decided.

They should:

* Return hard pass/fail
* Log decisions
* Be auditable

---

# 🧠 Architect Insight: Tools Define Control Boundaries

If a tool:

* Must always run → workflow
* May run conditionally → agent
* Must never be skipped → governance layer

This prevents chaos.

---

# 🏗 Step 3: Design Pattern for Your First Real Bank Agent

Here’s a mature pattern:

### Layer 1 – Deterministic Intake

* Validate required fields
* Basic rule validation

### Layer 2 – Agentic Evaluation

Agent decides:

* Which additional data to fetch
* Which models to run
* Whether more information is needed
* Whether alternative document suffices

### Layer 3 – Hard Compliance Gate

System enforces:

* AML must be checked
* KYC must be valid
* Regulatory rules must pass

### Layer 4 – Final Decision Engine

Deterministic aggregation:

* PD + LGD + Exposure
* Policy thresholds

Agent can recommend.
System decides.

That is boardroom-safe architecture.
